# v1.3.0 Plan — Seamless permissions & first-launch onboarding

Status: **planned** · Branch: `feature/v1.3.0-onboarding` · Merge to `main` only via approved PR.

## Why

Two first-run problems today:

1. **The permission flow is hostile.** At launch, a missing Screen Recording grant triggers a
   blocking alert and then **quits the app** ([AppDelegate.swift:112-139](../../Wardlume/AppDelegate.swift)).
   Accessibility + Input Monitoring are requested later via another modal alert mid–ward-activation,
   with "relaunch manually" guidance as plain text. Three permissions, three different moments,
   two dead ends.
2. **Opening the app appears to do nothing.** Wardlume is a menu-bar agent (`LSUIElement`), so
   launching it from /Applications just adds a status-bar icon — no window, no guidance.

## New behavior

On **launch** and on **reopen** (double-clicking the app in Finder while it's running):

| State | What the user sees |
|---|---|
| Any permission missing | **PermissionPilot onboarding wizard** — all three permissions in one guided window: explain → request → live status → relaunch handling |
| All permissions granted | **Settings window opens on the Overview pane** (ward status + checklist), instead of nothing |

The app never terminates itself over a missing permission anymore. Ward activation stays
**fail-closed** — the ward still refuses to arm without working permissions — but the "denied"
path now opens the wizard instead of a raw `NSAlert`.

## How — PermissionPilot integration

[PermissionPilot](https://github.com/arpitagarwal1301/PermissionPilot) (our in-house MIT package,
macOS 12+) provides everything the hand-rolled flow lacks:

- `PermissionManager(required: [.screenRecording, .accessibility, .inputMonitoring])` —
  live status (re-checks on app activation + fallback poll), request routing, System Settings
  deep-links, `relaunchSuggested`.
- `PermissionPilot.presentOnboarding(manager:configuration:onFinish:)` — the first-run wizard
  window; `hasCompletedOnboarding` persistence; `resetOnboarding()` for a "Run setup again" menu item.
- Config: Wardlume name/icon/reasons, teal tint, `colorScheme: .dark` to match the app.

### Sandbox compatibility (spike result — resolved)

Wardlume is sandboxed; PermissionPilot targets non-sandboxed apps. Verified from its sources:

- Its **detection/request calls are the same APIs Wardlume already ships sandboxed** —
  `AXIsProcessTrustedWithOptions`, `CGPreflight/CGRequestScreenCaptureAccess`; for Input
  Monitoring it uses `IOHIDCheckAccess`/`IOHIDRequestAccess` (same TCC service as our
  `CGPreflightListenEventAccess`). The README's "not usable sandboxed" caveat concerns
  capabilities like `AXUIElement` control — Wardlume's event-tap lock has worked sandboxed
  since v1.0, so it doesn't apply.
- **Real gap:** `PermissionManager.quitAndReopen()` shells out via `/bin/sh -c 'sleep 1; open …'`,
  which a sandboxed app cannot spawn. → upstream fix below.

### Upstream work in PermissionPilot (fix → release → integrate)

PermissionPilot is developed in its own repo/session. The sequencing is
**upstream-first**: fix and release there, then Wardlume consumes the released tag —
no app-side workarounds, no dependency on a dev branch.

1. **Fix in PermissionPilot** (integration testing against Wardlume found these):
   - `quitAndReopen()` quits without relaunching in sandboxed apps (`/bin/sh` helper
     is sandbox-forbidden) → relaunch via LaunchServices when sandboxed.
   - Sandbox *detection* must read the `com.apple.security.app-sandbox` entitlement
     from the process's code signature — the `APP_SANDBOX_CONTAINER_ID` env var is
     not reliably present (verified live) and misroutes to the shell path.
   - "Open the … list" walkthrough button called `request()` (the AX prompt is
     one-shot per app → silent no-op afterwards) → must deep-link the pane; the
     Accessibility request path also gains the pane fallback Screen Recording has.
   - Hardening: `OnboardingPresenter.front()`, `orderFrontRegardless()` on present,
     `relaunchAvailable` (hide dead Quit & Reopen), os.log on all relaunch paths.
   > Working, Wardlume-verified fixes already exist on branch
   > `fix/sandbox-safe-relaunch` in the local clone
   > (`~/arpitAgr/workspace/Developer/PermissionPilot`, commits e2a6ea2 · 4f6a753 ·
   > d1da729) — the PermissionPilot session can merge or reimplement them.
2. **Release `v0.2.0`** from the PermissionPilot repo (CHANGELOG, push, tag). The tag
   also delivers `colorScheme` / `showsWelcomeStep` / `showsDoneStep`, which `v0.1.0`
   lacks and Wardlume uses.
3. **Integrate**: Wardlume's temporary dev pin (local clone, fix branch) flips to
   `https://github.com/arpitagarwal1301/PermissionPilot.git`, `from: "0.2.0"`;
   fresh resolve + rebuild + smoke test (wizard, Accessibility deep-link,
   Quit & Reopen) — mandatory if upstream reimplemented rather than merged — then
   the feature branch is pushed and PR'd. **Do not merge while the pin still points
   at the local clone.**

## Risks & mitigations

- **Package is Swift 5.9 / pre-1.0** — public surface is `@MainActor`/`Sendable`, so Wardlume's
  MainActor-default target should consume it cleanly; pin the version, watch for API churn.
- **Screen Recording preflight is process-cached** — a fresh grant may read "missing" until
  relaunch; the wizard's relaunch banner + the sandbox-safe relauncher handle exactly this.
- **LSUIElement focus** — wizard window must be paired with `NSApp.activate(ignoringOtherApps:)`
  (existing pattern) and stay reachable from the status-bar menu.
- **Don't spam windows** — auto-open Settings only on launch/reopen events (there is no
  login-item auto-start today, so every launch is a deliberate user action).
- **TCC dev identity** — grants bind to code signature; test from a `/Applications` copy, reset
  state with `tccutil reset ScreenCapture|Accessibility|ListenEvent com.agarwal.wardlume.Wardlume`.

## Verification matrix (before PR)

1. Fresh machine state (all three reset) → launch shows wizard; grant all → wizard completes; ward arms.
2. Partial grants → wizard shows correct per-permission state; relaunch banner appears for Input Monitoring.
3. All granted → launch opens Settings on **Overview**; reopen-while-running does the same.
4. Missing perms at ⌘⇧L activation → wizard (not alert); ward still refuses to arm (fail-closed).
5. Regression: capture-loss/sleep teardown, watchdog, hotkeys, reaction packs untouched.

## Out of scope

Login-item auto-start, replacing the Settings checklist visuals with package components,
notarization (tracked in ROADMAP separately).
